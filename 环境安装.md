# 先说清楚:「环境」到底是什么

我们一直说"搭环境",但==环境==这个词本身得先定义清楚,否则后面所有组件都不知道挂在哪。

## 一个环境 = 三样东西的组合

> [!important] 定义
> 一个(Python)开发环境,在机制上 = ==一个特定的解释器 + 一组装好的、版本确定的依赖 + 它们被正确链接到的二进制/系统库==。三者绑成一套,共同决定"`import` 一个东西时,实际加载的是谁、哪个版本"。

具体拆开:

1. **解释器(interpreter)**:具体哪一个 `python.exe`、什么版本(3.10 还是 3.12)。版本决定了能装哪些预编译包(ABI),也决定了语法/标准库行为。
2. **依赖集合(packages)**:这个解释器对应的 ==site-packages== 里装了哪些库、各是什么版本。`import torch` 加载的就是这里面那一份。
3. **底层二进制 / 系统库**:很多包(torch、ffmpeg 绑定)不是纯 Python,而是==链接到一堆 .dll/.so==。环境还得保证这些底层库版本和上层 Python 包对得上(ABI 匹配)。

## 为什么"环境"必须是隔离的

如果所有项目共用一个 Python,就会遇到==依赖地狱(dependency hell)==:

> [!warning] 典型冲突
> - 项目 A 要 `numpy==1.24`,项目 B 要 `numpy>=2.0`,同一个 site-packages 里==只能存在一个版本==,必然有一个被踩坏。
> - 你为了跑 B 升级了 numpy,A 第二天就 import 报错或行为异常。
>
> 依赖图越大,这种==版本互相打架==的概率越高。LeRobot 依赖几十个包,几乎必然撞车。

==隔离== 的意思就是:==每个项目拥有自己独立的"解释器 + site-packages"==,各自一套版本,互不影响。这正是 conda 环境(或 venv)在做的事——给每个项目切出一份不共享的依赖空间。

## "切换环境"在机制上发生了什么

你电脑里同时存在多套这样的组合:

```
系统 Python        ← 装系统时自带,别动它
conda base 环境    ← conda 默认环境,也尽量别往里装项目依赖
conda lerobot 环境 ← 我们要建的、专门给这个项目的那套
```

`conda activate lerobot` 做的事,本质就是 ==把 PATH 指向 lerobot 这套==,于是此刻 `python` / `pip` / `import` 全部作用在它身上。所谓"在环境里装东西"、"在环境里跑代码",指的就是==当前 PATH 指向的那一套组合==。

> [!note] 一句话
> 后面 (1)~(7) 装的每个东西,都是在往 ==`lerobot` 这一套特定组合== 里填料,或在确认这套组合是否齐全可用。先有"环境=一套隔离的解释器+依赖"这个概念,后面才有挂靠点。

# 环境安装 · 组件机制与作用

## （1） conda / Miniconda

> [!info] 本次安装的具体版本(以截图为准)
> 安装包:==**Miniconda3 py312_24.7.1-0 (64-bit)**==
> - `py312` = base 环境自带 ==**Python 3.12**==(这是 base 的版本,**不是**我们项目要用的版本;我们后面会另建 ==Python 3.10== 的 `lerobot` 环境)。
> - `24.7.1-0` = ==Miniconda 安装器自身的版本号==。
> 下载地址 `Miniconda3-latest-Windows-x86_64.exe` 里的 `latest` 会随官方更新变,所以==实际装到的版本以安装界面标题为准==。

> [!warning] 安装界面两个关键选项(截图红框处)
> - **Install for**:选 ==Just Me (recommended)==,不需要管理员权限,装进当前用户目录。
> - **Advanced Options**:==**不要勾** "Add Miniconda3 to my PATH environment variable"==(官方标了 NOT recommended)。原因:把 conda 塞进系统 PATH 会和系统里其它 Python/程序==抢 `python`、`pip` 这些命令==,引发冲突。正确做法是用开始菜单的 ==Anaconda Prompt== 进 conda。
> - "Register Miniconda3 as my default Python 3.12" 勾不勾都行,勾上方便 VSCode/PyCharm 自动识别。

### 它到底是什么
conda 是一个 ==跨语言的二进制包管理器 + 环境管理器==,和 pip 有本质区别:

- **pip** 装的是 Python 包(wheel/sdist),只管 Python 层;遇到需要编译的 C/C++/Fortran 扩展,要么下预编译 wheel,要么在你机器上现场编译(经常因为缺编译器/缺系统库失败)。
- **conda** 装的是 ==已经编译好的二进制==,而且它能管 ==非 Python 的系统级依赖==(ffmpeg、CUDA runtime、MKL、glibc 之外的 so/dll 等)。它有自己的 ==依赖求解器(solver)==,在装包前先解一遍整个依赖图,保证版本互相兼容。

### conda 的"环境"在机制上是什么
一个 conda 环境 = ==一套独立的目录==(`envs/lerobot/`),里面有==自己的 Python 解释器、自己的 site-packages、自己的可执行文件目录==。`conda activate` 本质就是 ==改 PATH==(把这个环境的目录前置)+ 设几个环境变量。所以"激活"之后,`python`、`pip` 指向的都是这个环境里的那一份。

> [!note] 为什么 LeRobot 必须用它
> LeRobot 的依赖里有==大量带二进制扩展的东西==(torch、torchvision、ffmpeg、各种编码库)。这些用纯 pip 在 Windows 上装,版本对不齐、缺系统库、ABI 不匹配的概率非常高。
> conda 的价值就在于:==把这些二进制依赖和它们的系统库一起、版本协调地装进一个隔离目录==。环境之间互不污染,真出问题 `conda env remove` 重建即可,不动系统 Python。

## （2） channels(源)与换清华镜像

### channels 在机制上是什么
conda 装包时,会去 ==channels 列表里逐个查==该包及其依赖的可用版本,下载对应的二进制 + 元数据(`repodata.json`,记录每个包的依赖约束)。换源 = 换这些 ==repodata 和二进制的下载地址==。

### 为什么要换源(原因)
> [!important] 一句话原因
> conda 默认源在国外,==`repodata.json` 索引文件本身就几十 MB,国内拉取又慢又容易断==,经常卡在 "Solving environment" 或下载中途失败,导致环境装不全。
> 换成 ==清华 TUNA 镜像==(官方源的完整同步镜像)后:==下载快、稳==,能一次装齐。
> 注意:换源==只改变"从哪下",不改变"装出来的内容"==——功能完全一致,纯粹是速度和成功率的问题。

### 命令逐行讲解(可直接复制)
```shell
# 1. 先清空已有 channels 配置,避免和下面新加的源冲突/重复
#    (第一次装可能提示 key 不存在,属正常,忽略即可)
conda config --remove-key channels

# 2. 添加 conda 官方默认包源(清华镜像)
conda config --add channels https://mirrors.tuna.tsinghua.edu.cn/anaconda/pkgs/main/    # main:主仓库,最常用的包都在这
conda config --add channels https://mirrors.tuna.tsinghua.edu.cn/anaconda/pkgs/r/       # r:R 语言相关包
conda config --add channels https://mirrors.tuna.tsinghua.edu.cn/anaconda/pkgs/msys2/   # msys2:Windows 下的编译工具链

# 3. 添加常用第三方源(清华镜像)
conda config --add channels https://mirrors.tuna.tsinghua.edu.cn/anaconda/cloud/conda-forge/  # conda-forge:社区最大源,ffmpeg 等很多包在这更全更新
conda config --add channels https://mirrors.tuna.tsinghua.edu.cn/anaconda/cloud/msys2/        # 第三方 msys2 镜像
conda config --add channels https://mirrors.tuna.tsinghua.edu.cn/anaconda/cloud/bioconda/     # bioconda:生物信息相关
conda config --add channels https://mirrors.tuna.tsinghua.edu.cn/anaconda/cloud/menpo/        # menpo:计算机视觉相关
conda config --add channels https://mirrors.tuna.tsinghua.edu.cn/anaconda/cloud/pytorch/      # pytorch:PyTorch 官方源(注意:最后加=优先级最高)

# 4. 装包时显示每个包具体从哪个源下载,方便排查问题
conda config --set show_channel_urls yes

# 5. 清掉旧的索引缓存,强制让新源生效(否则可能还在用旧 repodata)
conda clean -i

# 6. 打印当前所有配置,验证上面的源是否都加成功
conda config --show-sources
```

| 命令 | ==作用== |
|---|---|
| `config --remove-key channels` | 清空旧源,防冲突 |
| `config --add channels <url>` | ==往源列表头部插一个源==(越后加优先级越高) |
| `config --set show_channel_urls yes` | 装包时显示来源 URL |
| `clean -i` | 清索引缓存,使新源生效 |
| `config --show-sources` | 查看/验证当前配置 |

> [!important] 两个容易踩的点
> - `--add channels` 是 ==往列表头部插==,越后加的优先级越高(所以脚本里==最后加的 `pytorch` 反而优先级最高==)。channel 优先级会影响 solver 选哪个源的包,源混用有时会解出奇怪的版本组合。
> - conda 源和 ==pip 源是两套独立配置==。LeRobot 最后是 `pip install -e` 装的,走 PyPI;==pip 不换源,这一步照样慢==。两边都要换:
>   ```shell
>   pip config set global.index-url https://pypi.tuna.tsinghua.edu.cn/simple
>   ```

## （3） 虚拟环境 `lerobot` + 锁定 python=3.10

### 为什么版本要钉死在 3.10
Python 版本不是随便选的,它==同时约束三样东西==:

1. **依赖能否安装**:很多带二进制扩展的包(torch、numpy、各种 sdk)是==按 Python 版本分别编译==的(cp310 这种 ABI tag)。Python 太新,可能根本没有对应的预编译 wheel。
2. **LeRobot 自己声明的约束**:它的 `pyproject.toml` 里有 `requires-python`,超出范围直接装不了。
3. **生态兼容性**:3.10 是当前这套机器人/深度学习栈==被验证过最稳==的区间;3.12/3.13 经常有某个依赖还没跟上。

> [!note] 所以这一步的实质
> 不是"建个环境"这么简单,而是 ==给整条依赖链钉一个所有包都能对齐的锚点==。版本选错,后面 `pip install` 会在解依赖时炸,而且报错信息往往很难直接看出根因。

## （4） 激活环境 `conda activate lerobot`

机制上面已经说了(改 PATH)。这里强调它的==实际后果==:

> [!danger] 没激活 = 操作错对象
> 不激活时,`python` / `pip` 指向 base 或系统 Python。这时候:
> - `pip install -e .` 会把 LeRobot ==装进错误的环境==;
> - `conda install ffmpeg` 如果你显式没指定环境,也可能装错地方;
> - 跑代码时 `import lerobot` 报 ModuleNotFound,因为你装在 A、运行在 B。
>
> 判断标准就一个:==命令行前缀是不是 `(lerobot)`==。这是"当前所有命令作用在哪个环境"的唯一可靠信号。

## （5） ffmpeg(conda 装,锁 7.1.1)

### 它在这套环境里的真实角色
ffmpeg 是 ==视频编解码的底层库 + 命令行工具==。LeRobot 处理视频不是自己实现编解码,而是==通过 PyAV / torchvision 这类 Python 绑定去调 ffmpeg 的库(libav*)==。链路是:

```
LeRobot → PyAV / torchvision.io → libavcodec/libavformat (ffmpeg) → 实际编解码
```

### 为什么它独立成一步、还要锁版本
- LeRobot 的数据集格式会把每个 episode 的摄像头流==编码成视频(常见 h264/AV1)再存==,读取时再解码成张量喂模型。==采集、回放、训练读数据,全部经过这条链==。
- 锁 7.1.1:上层绑定(PyAV/torchvision)对 ffmpeg 的 ==ABI / API 版本敏感==。版本不匹配的典型症状是:能 import,但一读/写视频就报编解码错误或段错误。所以这里的版本不是洁癖,是==绑定层的硬约束==。

> [!warning] 用 conda 装而不是系统装 ffmpeg
> 系统 PATH 里那个 ffmpeg 和 Python 绑定链接的 libav* ==不是一回事==。这里要的是==被 conda 环境内的 PyAV/torchvision 链接到的那一份==,所以必须 `conda install` 到环境里,而不是去官网下个 exe 丢进 PATH。

## （6） LeRobot 代码仓库(Seeed 版)+ `pip install -e ".[feetech]"`

### 这是项目本体
前面都是支撑,这个才是要用的框架:==数据采集、策略模型训练(ACT/Diffusion Policy 等)、推理部署==都在里面。Seeed fork 的意义是 ==针对 SO-ARM / feetech 这套硬件做了适配==(机型配置、舵机控制、标定脚本等),官方主线不一定覆盖你这套臂。

### `-e`(editable install)在机制上做了什么
普通 `pip install .` 是==把代码拷进 site-packages==,你改源码不生效得重装。
`-e` 是 ==editable/development 安装==:site-packages 里只放一个==指向你仓库目录的链接==(现代是 `__editable__` 的 `.pth` 文件),`import lerobot` 实际加载的就是==你 clone 出来的那份源码==。

> [!important] 为什么我们一定要 -e
> 你们后面要做的是 ==二次开发==:接超声器件、加触觉(mems 压阻/应变花)、改训练链路、打通全栈。这意味着==要频繁改 LeRobot 源码==。editable 模式下改完立刻生效,否则每改一行都得重装一次。

### `[feetech]` 是什么
`pyproject.toml` 里定义了 ==optional-dependencies(可选依赖组,extras)==。`pip install -e ".[feetech]"` 表示:装本体 + ==feetech 这一组额外依赖==。

- feetech(飞特)是 ==你机械臂舵机的品牌==,这组里是 ==驱动飞特舵机的 SDK 和相关依赖==(比如 `scservo_sdk`)。
- 换成别的硬件就是别的 extra(比如 `[dynamixel]`)。所以这个方括号里的名字 = =="我用的是哪套电机/硬件"==。

> [!caution] `".[feetech]"` 的引号
> 方括号在 shell 里是 glob 通配符。不加引号时,某些终端会尝试展开 `[feetech]`,导致 pip 收到的参数不对而报错。引号是为了==让 pip 拿到字面量字符串==。

## （7） 验证 import —— 按链路逐层确认

这四行对应==四条必须独立打通的链路==,任何一条断了,后面对应的功能就用不了:

| 命令 | 验证的链路 | 断了会怎样 |
|---|---|---|
| `import lerobot` | 框架本体安装是否成功(含 -e 链接是否生效) | 整个项目用不了 |
| `import scservo_sdk` | feetech extra 是否装上 → ==能否和舵机通信== | 连不上/控不了机械臂 |
| `import torch` | PyTorch(算力底座)是否就绪 | 无法训练/推理 |
| `torch.cuda.is_available()` | ==PyTorch 能否走 GPU==(torch 是否 CUDA 版 + 驱动是否就位) | `True` 用 GPU;`False` 只能 CPU,训练慢几十倍 |

> [!todo] 关于 cuda 返回 False
> 大概率是 ==pip 默认装了 CPU 版 torch==,或 CUDA/驱动没配。要训练就得==按显卡和 CUDA 版本重装 GPU 版 torch==(指定对应 index-url 的 cuxxx wheel)。这一步原教程没覆盖,属于额外配置。
